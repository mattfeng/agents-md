# SQLAlchemy and Alembic

## Database access
- Treat the devcontainer database connection as read-only. It may be used to inspect schema, metadata, constraints, indexes, and representative rows, but not to perform migrations.
- Do not run commands that mutate the database from the devcontainer, including `alembic upgrade`, `alembic downgrade`, `alembic stamp`, `metadata.create_all()`, `metadata.drop_all()`, or ad hoc DDL/DML.
- `alembic revision --autogenerate` is acceptable, but check all outputs for best practices in designing migrations.
- Design and edit migrations in the repo, then tell the user the exact command they must run outside the devcontainer to apply them.
- If a migration requires backfill, data cleanup, locking-sensitive DDL, or operational sequencing, document the manual steps and rollback considerations in the handoff.

## SQLAlchemy models
- Keep ORM mappings, table names, column types, nullability, defaults, constraints, indexes, and relationships aligned with the intended database schema.
- Prefer SQLAlchemy 2.x style APIs (`Mapped`, `mapped_column`, `select`) when the project already uses them.
- Be explicit about server-side behavior: use `server_default`, `server_onupdate`, database constraints, and database indexes when behavior must hold outside the Python process.
- Avoid hiding database I/O behind import-time side effects, global sessions, or model property access that unexpectedly emits queries.

## Sessions and transactions
- Keep transaction ownership clear at service boundaries. Do not commit inside low-level helpers unless the project already follows that pattern.
- Use `flush()` when code needs generated primary keys or constraint feedback before commit.
- Preserve existing sync versus async SQLAlchemy patterns; do not mix `Session` and `AsyncSession` styles in the same path.
- Watch for N+1 queries when changing relationships or serialization. Use loader options deliberately (`selectinload`, `joinedload`, `load_only`) and keep result cardinality in mind.

## Alembic migrations
- Prefer migrations that are deterministic, reviewable, and safe to run once. Avoid relying on local state, application imports with side effects, or live ORM sessions in revision files.
- Review autogenerated migrations carefully. Ensure constraints, indexes, enum changes, defaults, nullable changes, and foreign keys are intentional and reversible where practical.
- Include both `upgrade()` and `downgrade()` unless the project has an explicit irreversible migration policy. If downgrade is unsafe, raise a clear error and explain why in a comment.
- For non-trivial data migrations, separate schema changes from data changes when it reduces lock time, deployment risk, or rollback ambiguity.
- Use Alembic operations (`op.add_column`, `op.create_index`, `op.execute`, etc.) instead of application model metadata for migration logic.
- Account for database dialect behavior such as transactional DDL, enum alteration, concurrent index creation, column type rewrites, and SQLite batch operations.

## Handoff for manual execution
- When producing a migration, summarize the schema intent, generated revision file, and any assumptions about the current database state.
- Provide the exact manual command for the user to run outside the devcontainer, usually `alembic upgrade head`, plus any required environment variables or working directory.
- State whether the migration was written manually or based on autogenerate output, and whether it still needs to be tested against a writable database.
